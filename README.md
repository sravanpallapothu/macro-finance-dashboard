## 1. What This Does

This project automatically collects ~30 financial and macroeconomic indicators into a Google Sheet, exports them as static CSV files to a GitHub repository, and serves them through a single-page HTML dashboard on GitHub Pages.

**Four data frequencies are tracked:**

| Tab | Indicators | Update time |
|---|---|---|
| DailyData | Sensex, Nifty, USD/INR, Gold, Brent Crude, US yields, FII/DII flows | 6pm IST weekdays |
| WeeklyData | US gas price, US Fed gold stock, RBI forex reserves, M3, bank credit | Monday 9am + Saturday 9am IST |
| MonthlyData | India CPI, G-Sec yield, Central Bank Rate | 16th of each month |
| QuarterlyData | India Real GDP, property prices, current account, financial account | 16th of each month |

## 2. Architecture Overview

```
┌─────────────────────────────────────────────────────────────────┐
│  DATA SOURCES                                                   │
│  Yahoo Finance · FRED API · RBI Website · NSE India            │
└──────────────┬──────────────────────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────────────────────┐
│  GOOGLE SHEET  (auto-updated by Apps Script triggers)           │
│  DailyData · WeeklyData · MonthlyData · QuarterlyData          │
└──────────────┬──────────────────────────────────────────────────┘
               │  GitHub Actions runs daily at 7pm IST
               │  curl downloads each tab as CSV
               ▼
┌─────────────────────────────────────────────────────────────────┐
│  GITHUB REPOSITORY  (data/ folder)                              │
│  daily.csv · weekly.csv · monthly.csv · quarterly.csv          │
│  rbi_weekly.csv  (written by rbi_wss_updater.py)               │
└──────────────┬──────────────────────────────────────────────────┘
               │  GitHub Pages serves static files
               ▼
┌─────────────────────────────────────────────────────────────────┐
│  HTML DASHBOARD  (index.html)                                   │
│  Reads local CSV files · Instant load · No backend needed      │
└─────────────────────────────────────────────────────────────────┘
```

**Why static CSVs instead of live Google Sheets fetch?**

The dashboard originally fetched data directly from Google Sheets on every page load. This caused two problems: (1) Google Sheets export URLs require CORS headers that browsers block from external domains, causing the dashboard to get stuck loading indefinitely; (2) fetching from Google's servers on every page load was slow (5–30 seconds). The solution is to cache the data as static files in the same repository as the HTML. The browser fetches them from the same domain (GitHub Pages), which is instant and has no CORS issues. The data is at most one day old, which is acceptable since the underlying sheet only updates once daily anyway.

## 4. Part 2 — Yahoo Finance via Google Apps Script

### What is Google Apps Script?

Google Apps Script (GAS) is a JavaScript environment built into Google Workspace. It runs on Google's servers — not your computer — and can be attached directly to a Google Sheet. Scripts can be triggered on a schedule, making it ideal for daily data fetching with zero infrastructure.

Access it from your sheet via **Extensions → Apps Script**.

### How Yahoo Finance data is fetched

Yahoo Finance does not publish an official public API, but it exposes an undocumented JSON endpoint that Apps Script can call via `UrlFetchApp.fetch()` — the Apps Script equivalent of Python's `requests.get()`:

```javascript
function fetchYahooPrice(ticker) {
  const url =
    "https://query1.finance.yahoo.com/v8/finance/chart/" +
    encodeURIComponent(ticker) + "?interval=1d&range=1d";

  const response = UrlFetchApp.fetch(url, {
    method: "GET",
    headers: {
      "User-Agent": "Mozilla/5.0",
      "Accept":     "application/json",
    },
    muteHttpExceptions: true,
  });

  const json  = JSON.parse(response.getContentText());
  const price = json?.chart?.result?.[0]?.meta?.regularMarketPrice;
  return price;
}
```

The response is the same JSON object that Yahoo Finance's website uses internally. `regularMarketPrice` gives the last traded price — for Sensex and Nifty this equals the closing price when fetched after 3:30pm IST.

**Tickers used:**

| Indicator | Yahoo Finance Ticker |
|---|---|
| USD/INR exchange rate | `INR=X` |
| BSE Sensex | `^BSESN` |
| NSE Nifty 50 | `^NSEI` |
| Gold futures (USD/oz) | `GC=F` |
| Brent crude futures (USD/bbl) | `BZ=F` |
| US 10-year Treasury yield | `^TNX` |

### Historical backfill

For the initial population of historical data, the script uses Yahoo's historical endpoint with Unix timestamps as `period1` and `period2` parameters:

```javascript
function fetchYahooHistorical(ticker, period1, period2) {
  const url =
    "https://query1.finance.yahoo.com/v8/finance/chart/" +
    encodeURIComponent(ticker) +
    "?interval=1d&period1=" + period1 + "&period2=" + period2;

  const result  = JSON.parse(UrlFetchApp.fetch(url).getContentText())?.chart?.result?.[0];
  const timestamps = result?.timestamp || [];
  const closes     = result?.indicators?.quote?.[0]?.close || [];

  const series = {};
  for (let i = 0; i < timestamps.length; i++) {
    if (closes[i] != null) {
      series[unixToDateString(timestamps[i])] = closes[i];
    }
  }
  return series; // { "2026-01-02": 85762.01, ... }
}
```

The script builds a full date spine (every calendar day from the backfill start to today), then looks up each date in the Yahoo series. Missing dates (weekends, holidays) get -999.

### Setting up automated triggers

Triggers are created programmatically so you never need to configure them in the UI:

```javascript
function setupTriggers() {
  // Remove any existing managed triggers to avoid duplicates
  const managed = ["fetchDailyAndAppend", "fetchWeeklyAndAppend",
                   "fetchMonthlyAndAppend", "fetchQuarterlyAndAppend"];
  ScriptApp.getProjectTriggers()
    .filter(t => managed.includes(t.getHandlerFunction()))
    .forEach(t => ScriptApp.deleteTrigger(t));

  // Daily at 6pm — runs after Indian markets close at 3:30pm IST
  ScriptApp.newTrigger("fetchDailyAndAppend")
    .timeBased().everyDays(1).atHour(18).create();

  // Weekly on Mondays — picks up new FRED weekly observations
  ScriptApp.newTrigger("fetchWeeklyAndAppend")
    .timeBased().onWeekDay(ScriptApp.WeekDay.MONDAY).atHour(9).create();

  // Monthly on the 16th — picks up new FRED monthly/quarterly observations
  ScriptApp.newTrigger("fetchMonthlyAndAppend")
    .timeBased().onMonthDay(16).atHour(9).create();
  ScriptApp.newTrigger("fetchQuarterlyAndAppend")
    .timeBased().onMonthDay(16).atHour(9).create();
}
```

> **Critical:** Go to **Project Settings (gear icon) → Timezone** and set it to **Asia/Calcutta (IST)**. Without this, the `atHour(18)` trigger fires at 6pm UTC (11:30pm IST), not 6pm IST.

### Missing value convention

When a market is closed or a fetch fails, the script writes **-999** as a sentinel value rather than leaving the cell blank. This is used consistently across all data sources:

```javascript
const MISSING = -999;

// If Yahoo returns no data for a date (weekend, holiday):
row.push(MISSING);

// If a FRED observation is "." (FRED's own missing value indicator):
if (obs.value !== ".") series[obs.date] = parseFloat(obs.value);
```

-999 is filtered out before any chart or scorecard is rendered in the dashboard.

## 5. Part 3 — FRED API via Google Apps Script

### What is FRED?

FRED (Federal Reserve Economic Data) is the St. Louis Federal Reserve's database of approximately 800,000 economic time series from 100+ sources worldwide. It includes US monetary policy data (fed funds rate), international macro series (India CPI via OECD, India GDP via IMF), and commodity prices (gas via EIA). The API is free, well-documented, and stable.

### Getting a free FRED API key

1. Go to [fred.stlouisfed.org/useraccount/apikeys](https://fred.stlouisfed.org/useraccount/apikeys)
2. Create a free account (email and password — no credit card required)
3. Click **Request API Key** — the key appears immediately
4. It is a 32-character alphanumeric string, e.g. `abcdef1234567890abcdef1234567890`

### Storing the key securely in Apps Script

**Never paste your API key directly into the script code.** If you share or publish the script, your key would be exposed. Apps Script provides `PropertiesService` — an encrypted key-value store that persists separately from the script code and survives code replacements:

```javascript
function storeFredApiKey() {
  const ui  = SpreadsheetApp.getUi();
  const res = ui.prompt(
    "FRED API Key",
    "Paste your FRED API key here. It will be stored encrypted.",
    ui.ButtonSet.OK_CANCEL
  );
  if (res.getSelectedButton() !== ui.Button.OK) return;
  PropertiesService.getScriptProperties()
    .setProperty("FRED_API_KEY", res.getResponseText().trim());
  ui.alert("Key saved securely.");
}

function getFredApiKey() {
  const key = PropertiesService.getScriptProperties().getProperty("FRED_API_KEY");
  if (!key) throw new Error("Run storeFredApiKey() first.");
  return key;
}
```

Run `storeFredApiKey()` once from the Apps Script editor after pasting the script. You will never need to enter it again — even if you replace the script code later.

### Fetching FRED series data

```javascript
function fetchFredHistorical(seriesId, startDate) {
  const key = getFredApiKey();
  const url =
    "https://api.stlouisfed.org/fred/series/observations" +
    "?series_id="          + seriesId  +
    "&api_key="            + key       +
    "&file_type=json"                  +
    "&sort_order=asc"                  +
    "&observation_start="  + startDate; // e.g. "2024-01-01"

  const response = UrlFetchApp.fetch(url, { muteHttpExceptions: true });
  const json     = JSON.parse(response.getContentText());

  const series = {};
  for (const obs of (json.observations || [])) {
    if (obs.value !== ".") {  // FRED uses "." for missing observations
      series[obs.date] = parseFloat(obs.value);
    }
  }
  return series; // { "2024-01-01": 153.03, "2024-02-01": 153.37, ... }
}
```

**FRED series IDs used in this project:**

| Indicator | FRED ID | Via |
|---|---|---|
| US Fed Funds Rate (daily) | `DFF` | Federal Reserve |
| US Regular Gas Price (weekly) | `GASREGW` | EIA Monday survey |
| US Fed Gold Stock (weekly) | `WOFSRBGSA` | Federal Reserve H.4.1 |
| India CPI (Index, 2015=100) | `INDCPIALLMINMEI` | OECD |
| India Central Bank Rate | `IRSTCB01INM156N` | OECD |
| India 10yr G-Sec Yield | `INDIRLTLT01STM` | OECD |
| India Real GDP (Mn INR) | `NGDPRNSAXDCINQ` | IMF |
| India Property Price Index | `QINR628BIS` | BIS |
| India Current Account (USD Mn) | `INDB6BLTT02STSAQ` | OECD |
| India Financial Account (USD Mn) | `INDB6FATC01CXCUQ` | OECD |

> **Note on India series lag:** All India series on FRED are sourced via OECD, IMF, or BIS as intermediaries. They typically lag the original RBI/MoSPI release by 6–14 weeks. As of mid-2026, India CPI on FRED only goes up to early 2025.

## 6. Part 4 — RBI WSS Python Scraper

### Why Python instead of Apps Script?

The Reserve Bank of India's Weekly Statistical Supplement (WSS) is published on `rbi.org.in`. At first glance this appears to be a complex ASP.NET WebForms page requiring session cookies and hidden form tokens (`__VIEWSTATE`, `__EVENTVALIDATION`). However, investigation revealed that the Extract page accepts a `SelectedDate` parameter directly in the URL:

```
https://www.rbi.org.in/Scripts/BS_viewWssExtract.aspx?SelectedDate=6/6/2026
```

This returns the full data page without any form submission. Python's `requests` library can fetch this directly. `BeautifulSoup` with the `lxml` parser then extracts the data from the HTML.

Apps Script's `UrlFetchApp` could theoretically do the same, but the RBI page returns its entire content as a single large text blob inside one `<td>` element rather than as separate structured HTML tables. Parsing this with regex is easier in Python than in JavaScript, and the scraper runs as a standalone script on GitHub Actions where Python is readily available.

### What the WSS Extract page contains

The page at `/Scripts/BS_viewWssExtract.aspx` is a simplified version of the full 70-page Weekly Statistical Supplement. It contains four tables in a single text blob:

| Table | Content | Unit |
|---|---|---|
| Table 2 | Foreign Exchange Reserves: total, FCA, gold, SDRs, IMF position | USD millions |
| Table 4 | Scheduled Commercial Banks: aggregate deposits, bank credit | INR crore |
| Table 6 | Money Stock: M3, currency with public, time deposits | INR crore |
| Table 8 | Liquidity Operations: daily LAF injections and absorptions | INR crore |

> **What is NOT on the Extract page:** Repo rate, CRR, SLR, exchange rates (USD/INR), and bullion prices (gold/silver Mumbai market) are only in the full WSS PDF. The Extract is a summary subset.

### How the scraper works

```python
import requests
from bs4 import BeautifulSoup
import re

WSS_BASE_URL = "https://www.rbi.org.in/Scripts/BS_viewWssExtract.aspx"

def fetch_wss_text(date):
    """Fetch WSS data for a specific Friday date."""
    date_str = f"{date.month}/{date.day}/{date.year}"   # M/D/YYYY format
    url = f"{WSS_BASE_URL}?SelectedDate={date_str}"

    resp = requests.get(url, headers=HEADERS, timeout=30)
    soup = BeautifulSoup(resp.text, "lxml")

    # The data is in one large text blob inside a td element
    for td in soup.find_all(["td", "div", "p"]):
        text = td.get_text(" ", strip=True)
        if "Foreign Exchange Reserves" in text and "Total Reserves" in text:
            return text  # found it
    return None

def extract_row(text, date):
    """Extract indicator values using regex on the text blob."""
    result = {"Date": str(date)}

    # Format: "1 Total Reserves 6483678 682321 ..."
    # Columns: item_no  label    INR_Crore  USD_Mn  variation_cols...
    m = re.search(r'1\s+Total Reserves\s+[\d,]+\s+([\d,]+)', text)
    if m:
        usd_mn = float(m.group(1).replace(",", ""))
        result["Total Forex Reserves (USD Bn)"] = f"{usd_mn / 1000:.3f}"

    # Gold: "1.2 Gold 1069944 112600 ..."
    m = re.search(r'1\.2\s+Gold\s+[\d,]+\s+([\d,]+)', text)
    if m:
        result["Gold Reserves (USD Bn)"] = f"{float(m.group(1).replace(',','')) / 1000:.3f}"

    # M3: "M3 31453126 31023176 ..." (two columns: Mar 31 start | latest)
    m = re.search(r'\bM3\b\s+[\d,]+\s+([\d,]+)', text)
    if m:
        result["M3 Broad Money (INR Crore)"] = m.group(1).replace(",", "")

    return result
```

### Backfilling historical data

The scraper is designed to be idempotent — safe to run multiple times:

```python
BACKFILL_START = datetime.date(2026, 1, 1)
OUTPUT_FILE    = "data/rbi_weekly.csv"

def main():
    # Load dates already in the CSV (skip them)
    existing = load_existing_dates(OUTPUT_FILE)

    # Generate all Fridays from Jan 1 2026 to today
    fridays = all_fridays_since(BACKFILL_START)
    missing = [f for f in fridays if str(f) not in existing]

    print(f"Fridays total: {len(fridays)} | In CSV: {len(existing)} | To fetch: {len(missing)}")

    for i, friday in enumerate(missing, 1):
        print(f"[{i}/{len(missing)}] {friday}", end=" ")
        text = fetch_wss_text(friday)
        if text:
            row = extract_row(text, friday)
            write_rows(OUTPUT_FILE, [row])   # append, not overwrite
            print(f"-> {sum(1 for v in row.values() if v)}/{len(COLUMNS)} values")
        else:
            print("-> No data (holiday)")
        time.sleep(2)   # 2-second delay between requests
```

On the first run, this fetches ~22 Fridays (Jan–Jun 2026) and takes about a minute. On every subsequent run, it only fetches Fridays added since the last run — typically just one row per week.

## 7. Part 5 — GitHub Actions Automation

### What is GitHub Actions?

GitHub Actions is GitHub's built-in automation platform. It runs workflows (YAML files) on your repository in response to events — including scheduled cron jobs. It is completely free for public repositories and free up to 2,000 minutes per month for private repositories. The entire pipeline for this project uses approximately 3–4 minutes per day.

### Why download the Google Sheet as CSV via GitHub Actions?

Rather than fetching from Google Sheets on every dashboard page load, GitHub Actions downloads the sheet data once daily and commits it as static CSV files. The dashboard then reads these local files.

**Performance:** Static files served from GitHub Pages load in under 100 milliseconds. Fetching from Google Sheets live took 5–30 seconds and frequently triggered CORS errors.

**Reliability:** If Google Sheets is temporarily unavailable, the dashboard still works because the data was already committed to the repository.

**Simplicity:** No API keys, no authentication, no CORS headers needed. The browser fetches CSV files from the same domain as the HTML page.

### The main refresh workflow

There is a single workflow file — `.github/workflows/refresh_data.yml` — that handles everything: downloading the Google Sheet data, running the RBI scraper, and committing results back to the repository. You do not need any additional workflow files beyond this one.

It runs automatically every day at 7pm IST (1:30pm UTC), one hour after the Apps Script daily trigger fires at 6pm IST. This sequencing ensures that when GitHub Actions downloads the sheet, today's market data has already been written by Apps Script.

Here is the complete annotated workflow:

```yaml
name: Refresh Dashboard Data

on:
  schedule:
    - cron: "30 13 * * *"   # Runs at 1:30pm UTC every day = 7:00pm IST
                             # Cron format: minute hour day month weekday
                             # "* * *" at the end means every day of every month
  workflow_dispatch:         # Adds a "Run workflow" button in the GitHub Actions UI
                             # Useful for testing or forcing an immediate update
```

The `on:` block defines when the workflow runs. `schedule` uses standard Unix cron syntax. `workflow_dispatch` is a manual trigger — you can go to **Actions → Refresh Dashboard Data → Run workflow** at any time to trigger it immediately without waiting for the schedule.

```yaml
jobs:
  refresh:
    runs-on: ubuntu-latest   # GitHub spins up a fresh Ubuntu Linux VM for each run
    timeout-minutes: 15      # Kill the job if it runs longer than 15 minutes
                             # Prevents stuck jobs from consuming free minutes
```

GitHub Actions creates a brand new virtual machine (Ubuntu Linux) for every run. This VM has no memory of previous runs — it starts completely fresh every time. That's why the first step always checks out the repository code.

```yaml
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        # Clones your repository onto the VM so subsequent steps can read
        # and write files. Without this, the VM has no access to your repo.
        # actions/checkout@v4 is a pre-built action maintained by GitHub.
```

```yaml
      - name: Create data directory
        run: mkdir -p data
        # Creates the data/ folder if it doesn't exist.
        # -p means "no error if it already exists".
        # Needed on a fresh VM before the curl downloads try to write files there.
```

```yaml
      # ── Step 1: Download Google Sheet tabs as CSV ────────────────────────
      # Google Sheets exposes a public export endpoint for any sheet shared
      # as "Anyone with the link can view". No authentication needed.
      # URL format: /export?format=csv&gid=TAB_GID
      # Each tab has a unique GID visible in the URL when you click on it.

      - name: Download DailyData CSV
        run: |
          curl -sL -o data/daily.csv \
            "https://docs.google.com/spreadsheets/d/SHEET_ID/export?format=csv&gid=1210776088"
          echo "Daily: $(wc -l < data/daily.csv) rows"
          # curl flags:
          #   -s  = silent (no progress bar cluttering the log)
          #   -L  = follow redirects (Google redirects before serving the file)
          #   -o  = write output to this file path
          # The echo prints row count to the Actions log for verification.
```

The same pattern repeats for `weekly.csv` (gid=562712552), `monthly.csv` (gid=1304710388), and `quarterly.csv` (gid=1915292222). Each downloads the corresponding Google Sheet tab as a plain CSV file into the `data/` folder on the VM.

```yaml
      # ── Step 2: Run the RBI WSS Python scraper ──────────────────────────
      # The RBI scraper needs Python with specific libraries.
      # GitHub Actions VMs have Python pre-installed but not the libraries,
      # so we install them fresh on every run (takes ~10 seconds).

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install Python dependencies
        run: pip install requests beautifulsoup4 lxml
        # requests     — makes HTTP requests to rbi.org.in
        # beautifulsoup4 — parses the HTML response
        # lxml         — fast HTML parser used by BeautifulSoup

      - name: Run RBI WSS updater
        run: python rbi_wss_updater.py
        # This runs from the repository root where rbi_wss_updater.py lives.
        # The script:
        #   1. Reads data/rbi_weekly.csv to find dates already present
        #   2. Generates all Fridays from Jan 1 2026 to today
        #   3. For any Friday not yet in the CSV, fetches the RBI WSS page
        #   4. Extracts forex reserves, M3, bank credit etc. using regex
        #   5. Appends new rows to data/rbi_weekly.csv
        # On weekdays (Mon–Thu), no new Fridays exist so it exits immediately.
        # On Saturdays, it finds last Friday missing and fetches+appends it.
        # Safe to run multiple times — skips dates already in the CSV.
```

```yaml
      # ── Step 3: Commit all changed files back to the repository ─────────
      # The VM has the repo checked out and the data/ files have been updated.
      # We now commit those changes back to the main branch.

      - name: Commit updated data
        run: |
          git config user.name  "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          # Identity required for git commit — uses a standard bot account name.

          git add data/
          # Stages all files in the data/ folder (daily.csv, weekly.csv,
          # monthly.csv, quarterly.csv, rbi_weekly.csv).

          if git diff --staged --quiet; then
            echo "No changes — skipping commit"
          else
            git commit -m "Auto-refresh data $(date -u '+%Y-%m-%d %H:%M UTC')"
            git push
          fi
          # git diff --staged --quiet returns exit code 0 if nothing changed.
          # This check prevents empty "no-op" commits on weekends when markets
          # are closed and the sheet data hasn't changed.
          # When there are changes, the commit message includes the UTC timestamp
          # e.g. "Auto-refresh data 2026-06-11 13:30 UTC"
```

After `git push`, GitHub Pages automatically detects the new commit and redeploys the dashboard within ~30 seconds. The entire workflow typically completes in under 2 minutes.

### What happens on each day of the week

| Day | Google Sheet data | RBI scraper | Commit? |
|---|---|---|---|
| Monday–Thursday | New daily data downloaded | No new Friday → exits immediately | Yes (daily data changed) |
| Friday | New daily data downloaded | No new Friday yet (WSS published ~5pm) | Yes (daily data changed) |
| Saturday | No market data (weekend) | New Friday detected → fetches + appends | Yes (rbi_weekly.csv changed) |
| Sunday | No market data (weekend) | Friday already in CSV → exits immediately | No (nothing changed) |

Go to your repository → **Actions** tab → **Refresh Dashboard Data** in the left sidebar → **Run workflow** button → **Run workflow**. The job runs immediately and you can watch the logs in real time.

This is useful when:

### Enabling write permissions

### CSV parsing

Google Sheets exports numbers with commas (`"85,188.60"`), some columns have a `$` prefix (`$3.089`), and Apps Script occasionally writes dates as `M/D/YYYY HH:MM:SS` rather than `DD-MMM-YYYY`. Weekly data has alternating rows (gas price on Mondays, Fed gold on Wednesdays). The parser handles all of these:

```javascript
function cleanVal(raw) {
  let s = raw.toString().trim().replace(/^"+|"+$/g, '');
  // Detect -999 sentinel in all its variants
  if (['-$999','-$999.000','-999','-999.000'].includes(s)) return '__MISSING__';
  return s.replace(/^\$/, '').replace(/,/g, '');  // strip $ and commas
}

// Normalise any date format to YYYY-MM-DD
function toISO(s) {
  const iso = s.match(/^(\d{4}-\d{2}-\d{2})/);
  if (iso) return iso[1];  // already ISO, possibly with time suffix
  const dmy = s.match(/^(\d{2})-([A-Za-z]{3})-(\d{4})$/);
  if (dmy) {
    const m = {Jan:'01',Feb:'02',Mar:'03',Apr:'04',May:'05',Jun:'06',
               Jul:'07',Aug:'08',Sep:'09',Oct:'10',Nov:'11',Dec:'12'};
    return `${dmy[3]}-${m[dmy[2]]}-${dmy[1]}`;
  }
  // M/D/YYYY or M/D/YYYY HH:MM:SS from Apps Script
  const mdy = s.match(/^(\d{1,2})\/(\d{1,2})\/(\d{4})/);
  if (mdy) return `${mdy[3]}-${mdy[1].padStart(2,'0')}-${mdy[2].padStart(2,'0')}`;
  return s.slice(0, 10);
}
```

### Scorecards with sparklines

Each scorecard tile shows the latest non-missing value, the change from the previous non-missing value, and a 14-point sparkline. The sparkline is a pure SVG polyline — no chart library needed:

```javascript
function buildSparkline(rows, col, currentIdx) {
  // Collect up to 14 non-missing values ending at currentIdx
  const vals = [];
  for (let i = currentIdx; i >= Math.max(0, currentIdx-40) && vals.length < 14; i--) {
    const v = parseFloat(rows[i][col]);
    if (!isNaN(v) && v !== -999) vals.unshift(v);
  }
  if (vals.length < 2) return '';

  const W = 64, H = 24, pad = 2;
  const minV = Math.min(...vals), maxV = Math.max(...vals), range = maxV - minV || 1;
  const pts  = vals.map((v, i) => {
    const x = pad + (i / (vals.length-1)) * (W - pad*2);
    const y = H  - pad - ((v - minV) / range) * (H - pad*2);
    return `${x.toFixed(1)},${y.toFixed(1)}`;
  }).join(' ');

  return `<svg width="64" height="24">
    <polyline points="${pts}" fill="none" stroke="#4a90d9" stroke-width="1.5"
              stroke-linejoin="round" stroke-linecap="round"/>
  </svg>`;
}
```

## 11. Caveats & Known Limitations

**Missing values (-999):** Any cell showing -999 means data was unavailable — market holiday, exchange closed, or data not yet released. All charts and scorecards filter out -999 automatically. Never include -999 in any manual calculations.

**Yahoo Finance is unofficial:** Yahoo Finance does not publish a documented public API. The endpoint used (`query1.finance.yahoo.com/v8/finance/chart/`) is the one the Yahoo Finance website uses internally. Yahoo can change or remove it without notice. If daily data stops updating, check the Apps Script execution log for HTTP errors.

**FRED India series lag (6–14 weeks):** All India series on FRED are sourced via OECD, IMF, or BIS as intermediaries. They lag the original RBI/MoSPI release significantly. This dashboard is not a substitute for checking RBI or MoSPI directly for the most current CPI, GDP, or balance of payments figures.

**US Fed Gold Stock is statutory value, not market value:** `WOFSRBGSA` is valued at $42.22 per troy ounce — the statutory price frozen since 1973. The US holds ~8,133 tonnes and rarely changes its gold stock. To compute market value: `WOFSRBGSA × (Gold_USD_per_oz ÷ 42.22)`.

**Gold and Brent are futures prices:** GC=F and BZ=F are front-month futures contracts. They closely track spot prices but are not identical and do not represent formal daily settlement prices.

**RBI WSS Extract is a partial view:** The WSS Extract page does not contain repo rate, CRR, SLR, exchange rates, or Mumbai bullion prices. Those are only in the full WSS PDF which requires a more complex scraping approach.

**FII/DII data not yet active:** The `fii_dii_updater.py` workflow requires a Google Cloud service account. Until configured, FII and DII columns show -999.

**nsefin is Alpha software:** The `nsefin` library calls NSE India's internal (undocumented) API endpoints. NSE can change these without notice. This is the most fragile part of the pipeline.
